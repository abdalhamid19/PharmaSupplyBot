🎯 أنواع Queries المُولّدة

النظام يُنشئ 7 أنواع رئيسية من Queries لنفس الصنف:

1️⃣ Full Query (الاسم الكامل)

يحتوي على كل المعلومات المتاحة

الصنف: PANADOL ADVANCE 500 MG 24 TAB Query: "PANADOL ADVANCE 500 MG 24 TAB"
✅ الميزة: الأكثر دقة ووضوحاً
⚠️ العيب: قد لا يطابق إذا كانت بيانات Tawreed مختلفة قليلاً

2️⃣ Brand + Dosage Query

العلامة التجارية + الجرعة فقط

الصنف: PANADOL ADVANCE 500 MG 24 TAB Query: "PANADOL 500"
✅ الميزة: أبسط وأسرع، يجد نتائج أكثر
⚠️ العيب: قد يجد منتجات متعددة تحتاج فرز دقيق

3️⃣ Brand Only Query

العلامة التجارية فقط

الصنف: PANADOL ADVANCE 500 MG 24 TAB Query: "PANADOL"
⚠️ الميزة: يجد كل منتجات باندول
❌ العيب: نتائج كثيرة جداً، يحتاج فلترة قوية

4️⃣ Normalized Query

النص بعد التطبيع الكامل

الصنف: "Panadol-advance_500mg" Query: "PANADOL ADVANCE 500 MG"
✅ الميزة: يزيل الأخطاء الإملائية والرموز
✅ الميزة: يوحد الصيغ المختلفة

5️⃣ Without Quantity Query

بدون ذكر العدد

الصنف: PANADOL ADVANCE 500 MG 24 TAB Query: "PANADOL ADVANCE 500 MG TAB"
✅ الميزة: يجد المنتج حتى لو كان العدد مختلف
💡 ملاحظة: مفيد عندما يتوفر نفس المنتج بعبوات مختلفة

6️⃣ Brand + Form Query

العلامة + الشكل الصيدلاني

الصنف: PANADOL ADVANCE 500 MG 24 TAB Query: "PANADOL TAB"
💡 الاستخدام: مفيد عندما تكون الجرعات غير واضحة
⚠️ تحذير: يحتاج فلترة قوية جداً

7️⃣ Code Query (البحث بالكود)

كود الصنف من نظام الصيدلية

الكود: "12345" Query: "12345"
❌ نادراً ما ينجح: لأن كود الصيدلية يختلف عن كود Tawreed
💡 لكن: يُجرب كخيار أخير

🎮 استراتيجيات البحث المتقدمة

1. البحث بالفهرس (Brand Index Lookup)

الفهرس المُبني مسبقاً يربط كل علامة تجارية بمنتجاتها في قاعدة بيانات Tawreed. هذا يسمح بالبحث السريع جداً O(1) بدلاً من البحث الخطي.

مثال على الفهرس: brand_index = { "PAN": [0, 1, 2, 5, 8, 12], # منتجات تبدأ بـ PAN "PANA": [0, 1, 2, 5, 8], # منتجات تبدأ بـ PANA "PANAD": [0, 1, 2, 5], # منتجات تبدأ بـ PANAD "PANADO": [0, 1, 2], # منتجات تبدأ بـ PANADO "PANADOL": [0, 1, 2], # PANADOL بالضبط } Query: "PANADOL 500" ↓ استخراج Brand: "PANADOL" ↓ البحث في الفهرس: brand_index["PANADOL"] ↓ النتيجة: [0, 1, 2] (3 منتجات فقط بدل فحص 10,000+)
✅ المزايا:
  • سرعة فائقة: O(1) بدلاً من O(n)
  • يقلل عدد المنتجات المرشحة من 10,000+ إلى 3-50
  • يعمل حتى مع اختلافات بسيطة في الاسم

2. البحث بالمكونات (Component Index Lookup)

فهرس متقدم يربط مجموعات المكونات (Brand + Dosage، Brand + Volume، إلخ) بالمنتجات.

component_index = { ("brand", "PANADOL"): [0, 1, 2, 5, 8, 12], ("brand_dosage", "PANADOL", ("500",)): [0, 1, 2], ("brand_volume", "BRUFEN", "120"): [45, 46], ("brand_qty", "AMOXIL", "20"): [78, 79, 80], ("brand_flavor", "PANADOL", "ORANGE"): [10], } Query: "PANADOL 500 MG" ↓ المفاتيح المولدة: - ("brand", "PANADOL") - ("brand_dosage", "PANADOL", ("500",)) ↓ التقاطع: المنتجات الموجودة في كلا القائمتين ↓ النتيجة: [0, 1, 2] مع درجات تطابق أعلى
✅ المزايا:
  • دقة أعلى من Brand Index وحده
  • يعطي درجات تطابق معززة للتطابقات الدقيقة
  • يقلل False Positives

3. البحث الضبابي (Fuzzy Search)

عندما يفشل البحث بالفهرس، يُستخدم البحث الضبابي كـ fallback.

خوارزميات Fuzzy المستخدمة (بالترتيب): 1. token_set_ratio - يقارن مجموعة الكلمات الفريدة - مثال: "PANADOL 500" = "500 PANADOL" → 100% 2. token_sort_ratio - يرتب الكلمات قبل المقارنة - مثال: "ADVANCE PANADOL" = "PANADOL ADVANCE" → 100% 3. partial_token_sort_ratio - تطابق جزئي + ترتيب - مثال: "PANADOL 500" ⊂ "PANADOL ADVANCE 500 MG" → 95% الحد الأدنى: 80% (fuzzy_threshold)

⚠️ المشاكل والتحديات الحالية

❌ المشكلة 1: تسلسل Queries الثابت

المشكلة: النظام يجرب Queries بترتيب ثابت بغض النظر عن نوع المنتج
الترتيب الحالي (للجميع): 1. Full name 2. Brand + Dosage 3. Brand only 4. Normalized 5. Without quantity 6. Brand + Form 7. Code المشكلة: - للسوائل: Brand + Volume أهم من Brand + Dosage - للأصناف النادرة: Brand only قد يكون أفضل أولاً - للمستحضرات: الشكل أهم من الجرعة

مثال المشكلة:

الصنف: "BRUFEN SYRUP 100MG/5ML 120ML" الترتيب الحالي: ❌ Query 1: "BRUFEN SYRUP 100MG/5ML 120ML" → لا نتائج (دقيق جداً) ❌ Query 2: "BRUFEN 100" → نتائج خاطئة (أقراص!) ✅ Query 5: "BRUFEN 120ML" → ✅ الصحيح! المشكلة: استغرق 5 محاولات للوصول للنتيجة الصحيحة

❌ المشكلة 2: عدم استخدام سياق النتائج السابقة

المشكلة: كل Query مستقل ولا يتعلم من نتائج الـ Queries السابقة
السيناريو: Query 1: "PANADOL ADVANCE 500" → وجد 3 منتجات (كلها ADVANCE) Query 2: "PANADOL 500" → وجد 15 منتج (ADVANCE + EXTRA + عادي) المشكلة: النظام لا يستفيد من معلومة أن "ADVANCE موجود بالفعل" ويعيد البحث من الصفر في Query 2

❌ المشكلة 3: بطء البحث الضبابي الكامل

المشكلة: عند فشل الفهارس، يقارن مع جميع المنتجات (10,000+)
التعقيد الحالي: - Brand Index: O(1) - سريع جداً ✅ - Component Index: O(1) - سريع جداً ✅ - Fuzzy Fallback: O(n) - بطيء ❌ عند قاعدة بيانات 10,000 منتج: - Fuzzy على كل المنتجات = ~2-3 ثانية - إذا فشل Query الأول يجرب Query ثانٍ = +2-3 ثانية - المجموع للصنف الواحد = 5-10 ثوان مع 1000 صنف = 5000-10000 ثانية (ساعة ونصف!)

❌ المشكلة 4: عدم تحديد أولويات المرشحين

المشكلة: جميع المرشحين يُعاملون بنفس الأهمية
السيناريو: Query: "PANADOL 500" النتائج (15 منتج): 1. PANADOL 500 MG 24 TAB (Score: 92) 2. PANADOL EXTRA 500 MG (Score: 91) 3. PANADOL ADVANCE 500 MG (Score: 91) ... 15. PANADOL COLD FLU 500 MG (Score: 85) المشكلة: النظام يفحص جميع الـ 15 منتج بنفس التفصيل بدلاً من التركيز على الـ top 3 فقط

❌ المشكلة 5: تخزين مؤقت محدود

المشكلة: Cache يحفظ 16 query فقط، ثم يحذف الأقدم
السيناريو: - ملف يحتوي 100 صنف - كل صنف يجرب 3-5 queries - المجموع: 300-500 query مع Cache حجم 16: - فقط آخر 16 query محفوظة - 484 query الأخرى تُعاد من API معدل Cache Hit: 3% فقط!

✅ التحسين 4: Smart Caching Strategy

الفكرة: تخزين مؤقت متعدد المستويات

// هيكل Cache محسّن class SmartCache: def __init__(self): self.l1_cache = {} // آخر 50 query (سريع) self.l2_cache = {} // Queries شائعة (متوسط) self.frequency = {} // تتبع التكرار def get(self, query): // المستوى 1: الأحدث if query in self.l1_cache: self.frequency[query] += 1 return self.l1_cache[query] // المستوى 2: الشائع if query in self.l2_cache: self.promote_to_l1(query) return self.l2_cache[query] return None def put(self, query, results): self.l1_cache[query] = results self.frequency[query] = 1 // نقل الشائع إلى L2 if len(self.l1_cache) > 50: self.evict_to_l2() def evict_to_l2(self): // احتفظ بالأكثر تكراراً في L2 frequent = sorted( self.frequency.items(), key=lambda x: x[1], reverse=True )[:20] for query, _ in frequent: if query in self.l1_cache: self.l2_cache[query] = self.l1_cache[query]
🎯 الفائدة المتوقعة:
  • رفع معدل Cache Hit من 3% إلى 40-60%
  • الـ Queries الشائعة (مثل PANADOL) تبقى محفوظة دائماً
  • توفير آلاف طلبات API في الملفات الكبيرة

✅ التحسين 7: Early Stopping

def smart_search(item): confidence_threshold = 0.95 for query in generate_queries(item): results = search(query) // حساب ثقة النتيجة if results: best = results[0] confidence = calculate_confidence(best, item) // إيقاف مبكر إذا كانت الثقة عالية جداً if confidence >= confidence_threshold: log(f"Early stop: confidence={confidence}") return best // ✅ متأكدون، توقف هنا // استمر للـ query التالي // إذا وصلنا هنا، لم نجد تطابق عالي الثقة return best_from_all_results() // حساب الثقة def calculate_confidence(match, item): factors = [] // عوامل إيجابية factors.append(match.score / 100) // 0.0 - 1.0 factors.append(1.0 if exact_brand else 0.5) factors.append(1.0 if exact_dosage else 0.3) factors.append(1.0 if components_ok else 0.0) factors.append(0.8 if available else 0.4) // المتوسط المرجح weights = [0.3, 0.25, 0.25, 0.15, 0.05] confidence = sum(f * w for f, w in zip(factors, weights)) return confidence

  • نفس الدقة أو أعلى

🎓 الخلاصة

🔍 Query البحث - النقاط الرئيسية

✅ ما يعمل جيداً حالياً:

  • Brand Index: سريع جداً O(1) للبحث بالعلامة
  • Component Index: دقيق للمطابقات التفصيلية
  • Multiple Queries: يزيد فرص العثور على التطابق
  • Cache الأساسي: يوفر بعض الطلبات المكررة

⚠️ ما يحتاج تحسين:

  • Query Ordering: ثابت للجميع، يحتاج ديناميكية
  • Fuzzy Search: بطيء على قواعد البيانات الكبيرة
  • Cache: صغير جداً (16 فقط)، معدل hit منخفض
  • Candidate Selection: يفحص الكل بدل التركيز على الأفضل
  • Context Awareness: لا يتعلم من النتائج السابقة

🚀 التحسينات ذات الأولوية:

  1. Early Stopping - توقف عند اليقين (50% تسريع، سهل)
  2. Candidate Prioritization - افحص الأفضل أولاً (70% دقة، سهل)
  3. Fuzzy Pre-filtering - صفّي قبل المقارنة (15x أسرع، متوسط)
  4. Smart Caching - تخزين متعدد المستويات (60% hit rate، متوسط)
  5. Query Templates - قوالب حسب الفئة (40% أقل محاولات، متوسط)
