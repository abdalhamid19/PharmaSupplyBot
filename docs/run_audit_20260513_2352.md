## High Priority Issues

1. Accepted matches can miss `storeProductId`.

   Ten matched items in `order_item_summary_20260513_2352.csv` have
   `matched=True` but blank or `None` `matched_store_product_id`:

   - `91733` CHEMICETRIZINE 5 MG20 TAB
   - `73241` MINALAX 10 TAB
   - `57680` POTASSIUM CHLORIDE 5 ML
   - `74821` BETADINE ANTISEPTIC 60 ML SOLN. 10%
   - `28406` BEBELAC AR MILK
   - `16763` AMRIZOLE N SUPP
   - `79887` TELFAST DECONGESTANT 60/120MG 10 TABS.
   - `71903` NACTALIA 2 MILK 400 G
   - `61862` AMLODIPINE 5MG 30 TAB
   - `73564` CLEAREST 14 CAP

   Impact: a match without a Tawreed store product id is not reliably actionable
   for real cart insertion. The matching layer accepts by name/score but does not
   require an orderable identifier.

   Recommended fix: make actionability explicit. Before a `SearchMatch` can be
   used for ordering, require `storeProductId` or a known equivalent id from API
   or DOM parsing. If the API candidate lacks this id, fall back to DOM row
   extraction or mark the item as manual review.

2. Multiple accepted candidates exist for the same item.

   `match_only_summary_20260513_2352.csv` has 89 accepted candidate rows across
   28 accepted items. Several items have multiple accepted store ids:

   - `78379` NESTOGEN 3 MILK 200GM: store ids `2630924`, `2695428`
   - `91686` PANADOL EXTRA OPTI ZORB 24 F.C. TABS: store ids `2319620`, `2290018`
   - `21058` MIDODRINE 2.5 mg 20 TAB: five different store ids
   - `91304` panthenol 5% care cream 50 gm: ids include blank, `2176170`,
     `2490314`
   - `73508` GLUCOVANCE 500/5MG 30TAB: ids include blank and `2696100`

   Impact: the trace explains candidates, but the final summary hides ambiguity.
   This can select a valid name from a less suitable supplier, stock state, or
   malformed candidate.

   Recommended fix: add a deterministic tie-break report field:
   `winner_store_product_id`, `winner_available_quantity`, `winner_sale_price`,
   `winner_store_name`, and `tie_break_reason`. Reject or review when the best
   accepted candidates disagree by store id and the selected candidate has a blank id.

3. AI provider reliability is unstable.

   Latest run-control log `order_output_1778705566.log` includes 7 rate-limit
   warnings and 2 invalid JSON warnings. The AI trace records:

   - 54 API attempts with HTTP `200`
   - 4 API attempts with HTTP `429`
   - 2 parse failures from `big-pickle` with `error_code=invalid_json`

   Impact: item outcomes depend on fallback order and provider availability.
   This directly produced low-confidence or manual-review outcomes for some items.

   Recommended fix: persist provider health per run and skip attempts temporarily
   after repeated 429 or invalid JSON. Lower priority providers that repeatedly
   fail schema parsing.

4. Previous run had artifact schema failures.

   `artifacts/run-control/order/order_output_1778696048.log` contains 70 failed
   items caused by `dict contains fields not in fieldnames`. The current run no
   longer shows this failure, and tests pass, so this appears fixed in current
   artifact schema handling. Keep it in the regression list because it was a
   high-impact runtime failure.

## Code And Architecture Improvements

1. Enforce orderable candidate validation near `SearchMatch` creation.

   Relevant modules:
   - `src/core/product_matching.py`
   - `src/core/order_ai_records.py`
   - `src/tawreed/tawreed_product_search.py`
   - `src/core/order_run_artifact_rows.py`

   Add a helper such as `candidate_store_product_id(candidate)` and use it
   consistently in matching, AI record conversion, summaries, and cart actions.

2. Improve API/DOM fallback behavior.

   `src/tawreed/tawreed_product_search.py` prefers API candidates whenever the
   API returns any product list. If those candidates lack orderable ids, DOM
   parsing may still have the actionable row state. Treat "API rows without
   store ids" as partial data and enrich from DOM or fall back to DOM rows.

3. Add regression tests for blank `storeProductId`.

   Add tests that prove:
   - a candidate with no orderable id is not accepted for real ordering;
   - `match-only` can still log it as a diagnostic;
   - manual review rows include the candidate name and missing-id reason.

4. Add ambiguity tests for multiple accepted store ids.

   The current scoring can accept repeated query variants and different store ids.
   Add deterministic tie-break tests for stock, price, products count, supplier,
   and blank-id avoidance.

5. Reduce AI fallback noise.

   `src/core/drug_matching/ai_rotation*.py` and `verifier.py` should use latest
   run health when selecting attempts. Repeated 429 and invalid JSON should reduce
   provider priority inside the same run.

6. Split remaining large modules by responsibility.

   The audit still lists large files and long functions. Biggest files:
   - `src/core/drug_matching/trace_log.py`: 1,047 lines
   - `src/core/drug_matching/ai_steps.py`: 1,038 lines
   - `src/core/drug_matching/verifier.py`: 883 lines
   - `src/core/product_matching.py`: 776 lines
   - `src/tawreed/tawreed.py`: 771 lines

   This is baseline debt, not a current failing check, but it conflicts with
   the maintainability goals in `project_guidelines.md`.

7. Replace silent broad exceptions where practical.

   `rg` found many `except Exception` and several `pass` blocks in Tawreed
   integration modules. Some are acceptable for best-effort cleanup, but browser
   failures should record enough context to diagnose selector drift and network
   issues.

8. Keep Streamlit warnings out of unit-test output.

   Tests pass, but bare Streamlit warnings make CI logs harder to inspect.
   Isolate Streamlit import/render tests or filter known warnings.
