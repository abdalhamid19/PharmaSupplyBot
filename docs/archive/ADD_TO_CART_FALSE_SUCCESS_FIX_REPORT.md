# تقرير تشخيص وإصلاح مشكلة "added-to-cart الكاذب" في PharmaSupplyBot

- **التاريخ:** 2026-06-24
- **التشغيل محل المشكلة:** `artifacts/order/wardany/20260624_1359`
- **الأمر المُشغَّل:** `run.py order --profile wardany --execution-mode auto ...`
- **الأعراض:** التشغيل يقول `added-to-cart` ويخرج بنجاح (Exit code: 0)،
  لكن سلة موقع توريد لا تحتوي أي أصناف جديدة. خاصية match-only غير مُفعّلة.

## 1. الملخص التنفيذي (ابدأ من هنا)

السبب الجذري: **عقد الـ API المحفوظ كان يوجّه عملية "الإضافة للسلة" إلى نقطة
"قراءة السلة" بدل نقطة "الإضافة".**

- نقطة القراءة الصحيحة: `POST .../shopping/carts/items` → تُرجِع محتوى السلة.
- نقطة الإضافة الصحيحة: `POST .../shopping/carts/items/add` → تضيف فعلياً.

العقد المحفوظ في `state/tawreed_api_endpoints.json` كان يحمل
`add_to_cart_url = .../shopping/carts/items` (نقطة القراءة). عند الإرسال إليها
ترجع HTTP 200 ومعها قائمة السلة الحالية (التي كانت تحتوي 20 صنفاً سابقاً من
إضافات يدوية)، فتمر فحوصات النجاح ويُسجَّل `added-to-cart` — **بينما لم يُضَف أي
شيء فعلياً**.

السبب وراء حفظ النقطة الخاطئة: منطق اكتشاف العقد في
`tawreed_api_contract_merge.py` كان يعدّ السلسلة العامة `"carts/items"` علامةً
على نقطة الإضافة، وهي تطابق نقطة القراءة `.../carts/items` أيضاً. وبما أن طلب
القراءة يأتي قبل طلب الإضافة في التقاط المتصفح، اختار المنطق نقطة القراءة الخاطئة.

## 2. الأدلة الميدانية (Evidence)

### 2.1 التقاط المتصفح الحقيقي أثناء إضافة يدوية ناجحة
الملف: `artifacts/api_discovery/wardany/order_run_capture_20260623_203636.json`

|-----|-----------|--------------|
| `.../shopping/carts/items` (×3) | `{"mode":"error","data":{"productId":null,"storesList":null}}` | **قراءة/عرض** السلة |
| `.../shopping/carts/items/add` (×1) | `{"mode":"all","data":{"customerId":22823,"storeProductId":2066374,"quantity":1,"typeId":1}}` | **الإضافة الفعلية** |

هذا يثبت بشكل قاطع أن نقطة الإضافة هي `/carts/items/add` وأن `/carts/items` نقطة
قراءة فقط.

### 2.2 العقد المحفوظ كان خاطئاً
`state/tawreed_api_endpoints.json` (قبل الإصلاح):
```json
"add_to_cart_url": "https://api.tawreed.io/rest/v2/shopping/carts/items",
"add_to_cart_body": {"mode":"error","data":{"productId":null,"storesList":null}}
```
أي أنه يحمل نقطة وجسم **القراءة** مكان الإضافة.

### 2.3 سلوك نقطة القراءة عند الاستعلام
استعلام مباشر على `.../shopping/carts/items` أعاد `HTTP 200` مع `data` بطول 20
عنصراً (سلة فعلية فيها أصناف من إضافات سابقة). هذا يفسّر لماذا لم يكتشف الكود
الفشل: فحص النجاح كان يكتفي بأن `data` غير فارغة، والسلة أصلاً غير فارغة.

### 2.4 انتهاء صلاحية التوكن (سبب ثانوي مُستبعَد كمصدر للمشكلة)
- عمر التوكن: 1800 ثانية (30 دقيقة فقط): `iat=13:50:52`، `exp=14:20:52`.
- أثناء التشغيل (13:59) كان التوكن **صالحاً** → الـ API أعاد 200 → ظهر النجاح الكاذب.
- الـ 401 الذي يظهر الآن سببه انتهاء الصلاحية فقط، وليس سبب المشكلة الأصلية.
- لو كان التوكن منتهياً أثناء التشغيل، لظهر خطأ/تخطٍّ وليس `added-to-cart`.
- التشغيل القادم يستدعي تحديث المصادقة تلقائياً قبل طلب الـ API
  (`tawreed.py:262 _ensure_valid_auth()`)، فلن يتكرر الـ 401.

## 4. سلسلة الخطأ خطوة بخطوة

1. تشغيل سابق بالمتصفح التقط طلبات الـ API وحفظ العقد عبر منطق الدمج.
2. منطق الدمج عدّ `"carts/items"` علامةَ إضافة → التقط نقطة **القراءة** أولاً.
3. حُفِظ `add_to_cart_url = .../carts/items` في `state/tawreed_api_endpoints.json`.
4. في تشغيل الطلب (`auto`): `add_to_cart_url` غير فارغ → اعتُبر العقد مكتملاً → لا fallback.
5. لكل صنف: `add_to_cart()` أرسل جسم الإضافة الصحيح إلى نقطة **القراءة** الخاطئة.
6. نقطة القراءة أعادت `200` + قائمة السلة (20 صنفاً) → `_ensure_cart_item_added`
   مرّت لأن `data` غير فارغة → سُجِّل `added-to-cart` كذباً.
7. النتيجة: تقرير نجاح بلا أي إضافة فعلية للسلة.

## 6. التغييرات المُطبَّقة

### 6.1 `src/tawreed/tawreed_api_contract_merge.py`
علامات اكتشاف نقطة الإضافة: حُذِفت `"carts/items"` العامة، وصُحّحت
`"cart/items/add"` → `"carts/items/add"` وأُضيفت `"items/add"`. الآن نقطة
القراءة `/carts/items` لا تطابق أي علامة إضافة، بينما `/carts/items/add` تُطابَق
بصرف النظر عن ترتيب الالتقاط.

### 6.2 `src/tawreed/tawreed_api.py`
أُضيفت دالة `_is_trusted_add_to_cart_url(url)` التي ترفض النقطة المنتهية بـ
`carts/items` (نقطة القراءة) وتقبل `/carts/items/add`. وتُستخدم في:
- `contract_field_available("add_to_cart_url")` → تُرجع False لنقطة القراءة، فيحدث
  fallback للمتصفح في `auto` بدل النجاح الكاذب.
- `add_to_cart()` → ترفع `TawreedApiUnavailable` إذا كانت النقطة قراءةً أو فارغة.

هذا يجعل أي عقد قديم/خاطئ يُصحَّح ذاتياً بدل أن ينتج عنه نجاح كاذب.

### 6.3 `state/tawreed_api_endpoints.json`
صُحِّح `add_to_cart_url` إلى `.../shopping/carts/items/add` والجسم إلى الشكل
المُلتقَط `{"mode":"all","data":{customerId,storeProductId,quantity,typeId}}`،
ليعمل التشغيل القادم عبر الـ API فوراً.

### 6.4 اختبارات الانحدار
- `tests/test_tawreed_api_execution_mode.py`: التقاط نقطة القراءة **قبل** نقطة
  الإضافة يجب أن ينتج عنه عقد بنقطة `/carts/items/add`.
- `tests/test_tawreed_api.py`: عقد بنقطة القراءة يجعل
  `contract_field_available("add_to_cart_url")` ترجع False و`add_to_cart()` ترفع
  استثناءً.

### 6.5 توثيق
حُدِّث `PROJECT_MAP.md` ليعكس النقطة الموثوقة ومنطق الرفض.

- **خطر متبقٍّ وحيد:** شكل استجابة `/carts/items/add` عند النجاح غير مُلتقَط بعد.
  الفحص `_ensure_cart_item_added` يفترض أن الاستجابة الناجحة تحوي `data` غير فارغة.

  وهو فشل آمن (يُضاف الصنف عبر المتصفح) وليس نجاحاً كاذباً.
- **التحقق الحي المؤجّل (بسبب انتهاء التوكن الآن):** عند توفّر توكن صالح، شغّل أمر

  افتح السلة في الموقع للتأكد بصرياً.
