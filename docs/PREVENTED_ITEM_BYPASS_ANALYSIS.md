## 0. الملخص التنفيذي (Bottom Line)

السبب الجذري **ليس في منطق الطلب ولا في المطابقة**، بل **عيب في تطبيع الاسم
(name normalization) عند مقارنة الأصناف الممنوعة**:

> **الصنف الممنوع في الملف اسمه `BREVY 5 MG 10 F.C. TABS.` (بنقطة نهائية)،
> واسم الصنف في ملف الطلب `BREVY 5 MG 10 F.C. TABS` (بلا نقطة). دالة التطبيع
> `normalize_name` لا تزيل علامات الترقيم، فيصبح الاسمان غير متطابقين نصياً،
> فلا يُمنَع الصنف ويُطلَب.**

**الدليل القاطع (تشغيل الفلتر الفعلي):**
```
PREVENTED ENTRY: [('', "'BREVY 5 MG 10 F.C. TABS.'")]   ← الرمز فارغ + نقطة نهائية
  normalized prevented name: 'brevy 5 mg 10 f.c. tabs.'
ORDER item normalized name:  'brevy 5 mg 10 f.c. tabs'
SURVIVES FILTER (should be blocked=empty): [('83356', 'BREVY 5 MG 10 F.C. TABS')]
==> BLOCKED? False
```

**عامل مُفاقِم (ثانوي لكنه أصل المشكلة):** الصنف الممنوع في الملف **رمزه فارغ
(`None`)**، فالمنع يعتمد **حصرياً على تطابق الاسم النصي**. لا يوجد مسار احتياطي
عبر الرمز عندما يفشل تطابق الاسم.

**نطاق المشكلة:** ليست حالة فردية — **9 أصناف على الأقل** في ملف المنع تنتهي
أسماؤها بنقطة (BREVY, MICA, DIXVIT, IMATINIB, INJECTMOL, ROTADANSETRON×2,
CIPROFLOXMET, BABY RELIEF)، وكلها معرّضة لنفس التسرّب. و**116 من 312 صنفاً ممنوعاً
رمزها فارغ** ⟹ تعتمد كلياً على الاسم.

## 1. كيف يعمل منع الأصناف (خلفية ضرورية)

### 1.1 مسار التطبيق في الطلب

`load_regular_order_items` (`src/cli/commands/cli_order_items.py:161-176`):
```python
if has_prevented_filter and prevented_path is not None:
    prevented_items = load_prevented_items(prevented_path)   # سطر 174
    items = filter_prevented_order_items(items, prevented_items)  # سطر 175
```
يُحمَّل ملف المنع، ثم تُصفّى عناصر الطلب عبر `filter_prevented_order_items`.

### 1.2 تحميل ملف المنع

`load_prevented_items` (`src/core/ordering/prevented_items.py:67-98`) يقرأ
`data/input/prevented_items/drugprevented.xlsx`، بعمودين:
- `PREVENTED_CODE_COLUMN = "كود"` (سطر 23)
- `PREVENTED_NAME_COLUMN = "إسم الصنف"` (سطر 24)

كل صف يصبح `PreventedItem(code, name)`. **الاسم يُطبَّع عبر
`normalized_cell_text`** (يزيل المسافات الطرفية فقط، لا الترقيم).

### 1.3 منطق التصفية (بيت القصيد)

`filter_prevented_order_items` (`prevented_items.py:146-163`) يبني ثلاث مجموعات:
- `blocked_codes` — كل الرموز الممنوعة.
- `code_only_blocks` — رموز الأصناف الممنوعة التي لها **رمز بلا اسم**.
- `blocked_names` — كل الأسماء الممنوعة (بعد `normalize_name`).

ثم `_is_prevented_order_item` (`prevented_items.py:166-180`):
```python
item_code, item_name = normalized_key(item.code, item.name)
if item_name in blocked_names:              # (1) تطابق الاسم
    return True
if item_code and item_name:                 # (2) للصنف الذي له رمز+اسم:
    return item_code in code_only_blocks     #     يُمنع فقط لو الرمز ضمن "code_only_blocks"
if item_code:                                # (3) صنف له رمز فقط
    return item_code in blocked_codes
return False
```

> ⚠️ **ملاحظة معمارية حرجة:** إذا كان صنف الطلب له **رمز واسم معاً** (وهو الحال
> الشائع)، فإن الفحص عبر الرمز يقتصر على `code_only_blocks` (أي الأصناف الممنوعة
> التي **لا اسم لها**). فإذا كان الصنف الممنوع له اسم (كحالة BREVY)، فإن المطابقة
> بالرمز **لا تحدث إطلاقاً** — يبقى تطابق الاسم النصي (السطر 1) هو المسار الوحيد.

### 1.4 دالة التطبيع المعيبة

`normalize_name` (`src/core/identity/item_text.py:21-23`):
```python
def normalize_name(value: object) -> str:
    return " ".join(normalized_cell_text(value).lower().split())
```
تفعل **فقط**: تحويل لحروف صغيرة + توحيد المسافات. **لا تزيل النقاط، ولا علامات
الترقيم الطرفية، ولا تُوحّد الاختصارات (`F.C.`, `I.U.`).** لذا أي اختلاف ترقيم
دقيق بين ملف المنع وملف الطلب يكسر التطابق تماماً.

## 2. تشخيص الحالة بالأدلة (Evidence-Based Diagnosis)

### 2.1 محتوى ملف المنع الفعلي

### 2.2 محتوى ملف الطلب الفعلي

### 2.3 نتيجة التطبيع والمقارنة (تشغيل فعلي)

| المصدر | الاسم الخام | بعد `normalize_name` |

| ملف المنع | `BREVY 5 MG 10 F.C. TABS.` | `brevy 5 mg 10 f.c. tabs.` |

### 2.5 عامل تلوّث البيانات (تعارض الرمز 83356)

دليل إضافي على هشاشة الاعتماد على الرمز:

## 4. الحلول الممكنة (كل الخيارات مع المفاضلة)

### الحل أ — تشديد `normalize_name` لإزالة الترقيم الطرفي (✅ الموصى به للسبب #1)
- تعديل `normalize_name` (أو دالة نقية جديدة للمقارنة) لإزالة علامات الترقيم
  **الطرفية** ولإزالة المسافات المزدوجة، مع **الحذر الشديد**:
  - إزالة النقاط/الفواصل الطرفية من الاسم كاملاً وأطراف كل token.
  - **عدم** إزالة النقاط داخل الأرقام (`DIXVIT 10.000`) لتجنّب تشويه الجرعات.
- **الأثر:** `brevy ... tabs.` == `brevy ... tabs` ⟹ يُمنَع. يعالج الأصناف الـ9.
- **الخطر:** لو غيّرنا `normalize_name` المشتركة، قد تتأثر مسارات أخرى تستخدمها
  (المطابقة، remove_items). لذا الأأمن: **دالة تطبيع مقارنة مخصّصة للمنع** أو
  تطبيق التطبيع الأقوى على **الطرفين** (المنع + الطلب) بتناسق تام.

### الحل ب — إضافة مسار منع بالرمز موثوق (⚠️ مساعد محدود)
- توسيع `_is_prevented_order_item` ليطابق بالرمز حتى عندما يكون للصنف اسم.
- **الخطر:** الرموز ملغّمة (83356)، والأصناف الممنوعة أصلاً بلا رمز غالباً ⟹
  فائدة محدودة. **لا يحل حالة BREVY** لأن رمز المنع فارغ.

### الحل ج — مطابقة تقريبية/دلالية للأسماء الممنوعة (❌ مرفوض)
- استخدام fuzzy على الأسماء الممنوعة.
- **الخطر:** المنع قرار **حاسم** (يُسقط طلب المستخدم). المطابقة التقريبية قد
  تمنع أصنافاً غير مقصودة (false blocks) ⟹ غير آمن لقرار منع صارم.

### الحل د — تطبيع مصدر البيانات نفسه (✅ مكمّل، خارج الكود)
- تنظيف ملف `drugprevented.xlsx` من النقاط النهائية والمسافات المزدوجة.
- **الخطر:** حلّ يدوي هشّ يعود مع كل ملف جديد يرفعه المستخدم ⟹ يجب أن يكون
  الكود صامداً بذاته (الحل أ).

### الحل هـ — توحيد الاختصارات (F.C./I.U./I.V.) (✅ تحسين اختياري)
- تطبيع الاختصارات الطبية بحذف النقاط الداخلية بتناسق في الطرفين.
- **الأثر:** يزيد الصمود أمام اختلافات مثل `F.C.TABS` vs `F.C. TABS`.

## 7. مخاطر وملاحظات (Risks & Notes)

1. **`normalize_name` مشتركة:** تُستخدم خارج المنع أيضاً ⟹ الأأمن دالة تطبيع
   مقارنة **مخصّصة للمنع** بدل تغيير المشتركة، أو تطبيق حذر مع تغطية اختبارية
   واسعة لتجنّب الانحدار في المطابقة/remove_items.
2. **النقطة العشرية:** `DIXVIT 10.000` — إزالة النقاط العمياء تخلط `10.000` بـ
   `10000` وقد تسبب منعاً خاطئاً؛ يجب حصر الإزالة في الترقيم **الطرفي/المحيط
   بالكلمات** لا داخل الأرقام.
3. **المسافة المزدوجة:** `IMATINIB  400` فيها مسافتان — `split()` يعالجها أصلاً،
   لكن يجب التأكد بالاختبار.
4. **الرموز ملغّمة:** الرمز 83356 يشير لمنتجين ⟹ لا يُعتمَد المنع بالرمز وحده؛
   الاسم المطبّع بدقة هو الأوثق.
5. **قرار المنع حاسم:** أي تطبيع أقوى يجب أن يُطبَّق **بتناسق تام على الطرفين**
   حتى لا يتحول لمطابقة فضفاضة تمنع أصنافاً بريئة.
6. **لا تغيير سلوكي غير مقصود:** كل بند مغطّى باختبار حماية، والتطبيع الأقوى
   محدود بالترقيم الطرفي فقط، التزاماً بـ `project_guidelines.md`
   و`starting_prompt.md`.
