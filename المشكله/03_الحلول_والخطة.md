# الحلول الممكنة والخطة الكاملة

## الهدف القابل للتحقق

عند تشغيل `matched-only` على منتج له مخازن متعددة، يجب أن يسجل الملخص المخزن الذي كان سيختاره مسار الطلب الفعلي حسب `warehouse_strategy.mode`، بدون إضافة أي شيء للسلة.

بالنسبة للحالة المعطاة:

```text
input: ZINC OLIVE CREAM 75 GM
mode: max_discount
stores: الهادي 32%, روما 24%, ...
expected winner_store_name: شركه الهادي فارم (الجيزه)
expected winner_discount: 32%
```

## الحل الأول: قراءة تفاصيل المخازن في matched-only وتسجيل اختيار افتراضي

الحالة: تم تنفيذه.

الفكرة:

- بعد قبول المنتج في `matched-only`، لا نضيف للسلة.
- إذا كان المنتج multi-store، نفتح تفاصيل المخازن للقراءة فقط.
- نستدعي نفس دالة الاختيار المستخدمة في الطلب الحقيقي:

```python
choose_next_store_for_remaining_quantity(...)
```

- نسجل فقط:

```python
bot.last_selected_store_name
bot.last_selected_discount_percent
```

المميزات:

- أصغر تغيير مباشر.
- يعيد استخدام منطق الاختيار الحالي بدل نسخ قواعد جديدة.
- يجعل artifact `matched-only` مطابقًا لما كان سيحدث عند الطلب الفعلي.
- لا يغير منطق المطابقة ولا المراجعة اليدوية.
- لا يلمس السلة.

العيوب:

- `matched-only` سيجري طلبًا إضافيًا لتفاصيل المخازن للمنتجات متعددة المخازن.
- إذا فشل فتح التفاصيل لأي سبب UI/API، يعود إلى بيانات المرشح بدل إسقاط الصنف، وهذا قرار مقصود حتى لا يتحول match-only إلى مسار هش.

## الحل الثاني: جعل matched-only يطبع فقط بيانات المنتج ولا يطبع winner_store

الحالة: لم يتم اختياره.

الفكرة:

- نمنع عرض `winner_store_name` و `winner_discount` في match-only لأنه لم يطلب فعليًا.

المميزات:

- لا توجد طلبات إضافية.
- يمنع عرض معلومة مضللة.

العيوب:

- لا يحقق طلب المستخدم بمعرفة أعلى خصم.
- يقلل فائدة `matched-only` في مراجعة قرارات الشراء قبل التنفيذ.
- لا يساعد في اكتشاف هل `max_discount` سيختار المخزن الصحيح أم لا.

## الحل الثالث: تعديل matching نفسه ليختار المرشح الأعلى خصمًا من نتائج البحث

الحالة: مرفوض لهذه المشكلة.

الفكرة:

- نجعل `explain_best_product_match` يفضل أعلى خصم أثناء اختيار المنتج.

العيوب:

- يخلط بين اختيار المنتج واختيار المخزن.
- قد يغير سلوك المطابقة بشكل خطير، خصوصًا مع أسماء منتجات متشابهة.
- تفاصيل المخازن قد لا تكون موجودة في نتائج البحث أصلًا.
- يناقض فصل المسؤوليات في `project_guidelines.md`.

## الحل الرابع: إجبار المراجعة اليدوية على حفظ storeProductId محدد لكل مخزن

الحالة: غير مناسب الآن.

الفكرة:

- عند حفظ manual review، نخزن مخزنًا بعينه بدل المنتج فقط.

العيوب:

- المخزون والخصومات تتغير يوميًا، وحفظ مخزن ثابت قد يصبح أسوأ من اختيار أعلى خصم الحالي.
- المشكلة الحالية ليست أن المنتج خطأ، بل أن المخزن لم يتم اختياره من التفاصيل في `matched-only`.

## الحل الخامس: إضافة عمودين منفصلين للتمييز بين search-row وselected-store

الحالة: حل مستقبلي مفيد، لكنه أكبر من المطلوب.

الفكرة:

- نضيف مثلًا:

```text
search_row_store_name
search_row_discount
selected_store_name
selected_discount_percent
```

المميزات:

- يقلل الالتباس في التقارير.
- يساعد في تحليل فروق صف البحث وتفاصيل المخازن.

العيوب:

- يغير schema الأرتيفاكت ويحتاج اختبارات وتحديثات UI محتملة.
- ليس ضروريًا لإصلاح الخطأ الحالي.

## الخطة التي تم اتباعها

1. قراءة تعليمات المشروع.
2. فحص حالة Git لتحديد التغييرات الموجودة مسبقًا.
3. البحث عن كل مسارات `discount`, `warehouse`, `manual review`, `matched-only`.
4. تتبع مسار `matched-only` في Browser.
5. تتبع مسار `add-to-cart` للمقارنة.
6. تتبع بناء حقول `winner_store_name` و `winner_discount` في الأرتيفاكت.
7. تتبع مسار API match-only للتأكد من أن الإصلاح لا يقتصر على Browser.
8. تنفيذ إصلاح صغير يقرأ تفاصيل المخازن في `matched-only` فقط لتسجيل metadata الاختيار.
9. إضافة اختبار Browser يحاكي حالة روما `20%` في صف البحث والهادي `32%` في التفاصيل.
10. إضافة اختبار API لنفس الفكرة.
11. تشغيل الاختبارات المركزة.
12. تشغيل التحقق العام للمشروع.
13. تحديث تقرير Markdown بنتائج التحقيق والاختبارات.
14. مراجعة diff ثم commit للتغييرات المقصودة فقط.

## الملفات التي تغيرت ولماذا

### `src/tawreed/products/tawreed_match_only_metadata.py`

أضيف ملف جديد يحتوي دوال:

```python
record_match_only_store_metadata(...)
match_only_store_choice(...)
```

وظيفتها:

- تسجيل بيانات صف البحث أولًا كـ fallback.
- إذا المنتج متعدد المخازن، تفتح تفاصيل المخازن وتختار حسب الاستراتيجية.
- تسجل المخزن المختار فعليًا في `bot.last_selected_*`.

### `src/tawreed/products/tawreed_products_flow.py`

بقي منطق الطلب الحقيقي كما هو، وتم الاعتماد على دواله القائمة مثل `matched_product_row`, `open_stores_dialog`, `_wh_mode`, `_min_disc`, و `_preferred_warehouses` من ملف metadata الجديد بدل نسخ منطق جديد.

### `src/tawreed/order/tawreed_order_processing.py`

تم تعديل `match_item_only` ليستدعي تسجيل بيانات المخزن بعد قبول المطابقة:

```python
record_match_only_store_metadata(self.bot, page, match, active_query)
```

### `src/tawreed/api/tawreed_api_match_only_metadata.py`

أضيف ملف جديد لمسار API يحتوي:

```python
record_api_match_only_store_metadata(...)
api_match_only_store_choice(...)
```

وظيفته نفس وظيفة مسار Browser لكن عبر `api.get_store_details(...)`.

### `src/tawreed/api/tawreed_api_flow_main.py`

تم تعديل API match-only بنفس الفكرة:

```python
record_api_match_only_store_metadata(bot, api, match)
```

### `src/tawreed/api/tawreed_api_flow.py`

تمت إعادة تصدير helper الجديد للاختبارات والاتساق مع نمط الملف الحالي.

### `tests/tawreed/store/test_tawreed_bot.py`

أضيف اختبار يثبت أن Browser match-only يسجل `الهادي / 32%` بدل `روما / 20%` عند `max_discount`.

### `tests/tawreed/api/test_tawreed_api_execution_mode.py`

أضيف اختبار مماثل لمسار API match-only.

## المخاطر المتبقية

- إذا فشل endpoint أو dialog الخاص بتفاصيل المخازن وقت التشغيل، سيبقى التقرير على fallback بدل إسقاط الصنف. هذا مقصود لتجنب تحويل match-only إلى عملية أضعف من المطابقة نفسها.
- إذا كانت تفاصيل المخازن تحتوي أكثر من product identity بنفس الاسم وسعر عام مختلف، فالأفضل مستقبلًا إضافة أعمدة تشخيص توضح `productId`, `storeProductId`, `retailPrice`, `salePrice` للمخزن المختار.
- إذا لم يكن `warehouse_mode=max_discount` مفعلًا في التشغيل الحقيقي، فلن يختار النظام أعلى خصم، وهذا سلوك إعدادات وليس bug.
