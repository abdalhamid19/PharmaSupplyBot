## دليل المشكلة من بيانات المستخدم

القيمة المسجلة في الملخص:

```text
winner_store_name=شركه روما فارما (الجيزه)
winner_discount=20%
winner_sale_price=95
winner_available_quantity=10
tie_break_reason=Approved by saved manual review (Name match).
```

بينما تفاصيل المخازن التي قدمتها تعرض:

```text
شركه الهادي فارم (الجيزه)      32.00   97.92 144.00
شركه التحرير (الجيزه)          25.00   108.00 144.00
شركه مصر مديكال (الجيزه)       25.00   108.00 144.00
شركه روما فارما (الجيزه)       24.00   109.44 144.00
```

الاستنتاج من هذه المقارنة:

- `20%` ليست أعلى خصم.
- حتى مخزن روما نفسه في قائمة التفاصيل ليس `20%` بل `24%`.
- القيمة `95` لا تتوافق حسابيًا مع السعر العام `144` وخصم `20%`، لأن خصم `20%` من `144` يساوي `115.2` تقريبًا.
- إذًا الملخص كان يأخذ بيانات من مصدر مختصر مختلف عن قائمة تفاصيل المخازن.

## مسار match-only قبل الإصلاح

في `src/tawreed/order/tawreed_order_processing.py` كانت الدالة:

```python
def match_item_only(self, page: Page, item: Item) -> None:
    self.ensure_match_only_surface(page)
    match, _ = require_product_match(self.bot, page, item, require_available=False)
    self.bot.log(f"Match-only accepted {item.code} / {item.name}: {match.query}")
```

هذا المسار كان يفعل التالي فقط:

- يتأكد أن صفحة البحث جاهزة.
- يبحث عن المنتج ويقبل المطابقة.
- لا يفتح نافذة تفاصيل المخازن.
- لا يستدعي اختيار المخزن حسب `max_discount`.
- يترك `last_selected_store_name` و `last_selected_discount_percent` فارغين أو معتمدين على بيانات المرشح المختصر لاحقًا في artifact row.

## كيف يكتب الملخص حقول winner

في `src/core/ordering/order_winner_fields.py`:

```python
def _extract_store_info(candidate: dict, summary) -> tuple[str, str]:
    store_name = candidate.get("storeName", "")
    discount = ""
    if summary:
        if getattr(summary, "selected_store_name", ""):
            store_name = summary.selected_store_name
        discount = getattr(summary, "selected_discount_percent", "")
    return store_name, discount
```

المعنى:

- إذا كان `summary.selected_store_name` موجودًا، يستخدمه كحقيقة اختيار المخزن.
- إذا لم يكن موجودًا، يرجع إلى `candidate.get("storeName")` من صف البحث المختصر.
- `winner_discount` يأخذ `summary.selected_discount_percent`، وإذا لم يتم تسجيله بدقة فقد يظهر الخصم من صف البحث أو يبقى ناقصًا حسب مسار بناء الصف.

قبل الإصلاح، `matched-only` لم يكن يسجل `selected_store_name` و `selected_discount_percent` من تفاصيل المخازن، لذلك أصبح الملخص معرضًا لبيانات صف البحث المختصر.

## منطق اختيار أعلى خصم

في `src/tawreed/store/tawreed_store_selection.py`:

```python
if mode == "max_discount":
    return max(choices, key=lambda c: (tier(c), -c.priority_score, c.available_quantity))
```

هذا يعني أن الاختيار في وضع `max_discount` يعتمد على:

- أعلى خصم بعد تقريب بسيط لتجميع الخصومات القريبة جدًا.
- أولوية المخزن المفضلة عند تعادل الخصم تقريبًا.
- كمية المخزن عند استمرار التعادل.

لو وصلت قائمة المخازن التي قدمتها إلى هذه الدالة، فالاختيار الطبيعي يكون `شركه الهادي فارم (الجيزه)` بخصم `32%`.

## دور المراجعة اليدوية

السبب المسجل:

`Approved by saved manual review (Name match).`

مصدره في `src/core/manual_review/manual_review_helpers.py`:

```python
return MatchDecision(match, [], "Approved by saved manual review (Name match).")
```

هذا يحدث عندما يجد النظام مرشحًا اسمه مطابق للاسم المحفوظ في المراجعة اليدوية.

المراجعة اليدوية هنا تقرر المنتج، وليس المخزن. لذلك وجود هذا السبب لا يبرر اختيار خصم `20%`، لكنه يشرح لماذا كانت درجة المطابقة `999` ولماذا تم قبول المنتج مباشرة.

## لماذا لم أعتمد على Screenshot أو Browser Live

الأدوات المتاحة هنا لا تتضمن أداة Browser/Screenshot مباشرة. كما أن فتح Tawreed حيًا قد يتطلب جلسة مصادقة صالحة وقد يلامس بيئة خارجية حساسة. لذلك اعتمد التحقيق على:

- كود مسار Browser.
- كود مسار API.
- artifacts والملخصات المتاحة محليًا.
- اختبارات وحدة تحاكي قائمة المخازن التي ذكرتها.

هذا كافٍ لعزل سبب الخلل لأن المشكلة في ترتيب خطوات الكود: `matched-only` لم يكن يصل أصلًا إلى خطوة تفاصيل المخازن.
