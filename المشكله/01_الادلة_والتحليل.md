## مسار تشغيل ORDER RUN من Streamlit

مسار التشغيل من الواجهة:

1. `src/ui/order/streamlit_order.py::render_order_tab`
2. `src/ui/order/streamlit_order.py::run_order_submission`
3. `src/ui/order/streamlit_order_process.py::start_order_process`
4. `src/ui/views/streamlit_process.py::start_cli_subprocess`
5. تشغيل CLI في الخلفية: `python run.py order ...`

قبل الإصلاح، `run_order_submission` كان يفعل التالي:

1. يحضر مسار الـ summary.
2. يبني أمر CLI.
3. يبدأ subprocess.
4. يضع العملية في `st.session_state["order_process"]`.
5. ينفذ `st.rerun()`.

لكن لم يكن هناك أي تحقق من أن نفس النموذج بنفس ملف Excel بدأ قبل ذلك بالفعل.

## مسار انتهاء العملية من Streamlit

في `src/ui/order/streamlit_order_process.py`:

- إذا كان `process.poll()` يرجع كود خروج، يتم عرض نتيجة العملية.
- بعدها يتم تنفيذ:

```python
st.session_state.pop("order_process", None)
```

هذا السطر صحيح بحد ذاته، لأنه ينظف حالة العملية المنتهية، لكنه كان يترك الواجهة بدون أي ذاكرة تمنع تكرار نفس التشغيل.

## دليل من artifacts

تم فحص لوجات كثيرة داخل:

`artifacts/run-control/order/`

لوحظ وجود تشغيلات متعددة جدًا لنفس profile `wardany`، وأمثلة على تشغيلات متتابعة كانت تبدأ Run جديدًا داخل مجلدات artifact جديدة:

```text
[wardany] Artifact run: artifacts\order\wardany\20260622_1856
[wardany] Launching 4 parallel item workers...
```

كما وُجدت تشغيلات تنتهي بإضافة أصناف ثم بعدها توجد تشغيلات جديدة في ملفات لاحقة:

```text
[wardany] Items added to cart. Final order submission is disabled.
[wardany] order completed with Tawreed API backend.
```

هذا لا يثبت وحده أن Streamlit أعاد التشغيل تلقائيًا، لكنه يدعم أن سلوك بدء runs جديدة بنفس واجهة التحكم كان متكررًا، وأن غياب حماية idempotency في الواجهة كان ثغرة حقيقية.

## فحص مسار قراءة Excel

القراءة تتم في:

`src/core/utils/excel.py`

الدوال المهمة:

- `load_items_from_excel`
- `_read_item_rows`
- `_limited_items`

القراءة تعتمد على generator وتبدأ مرة واحدة من الصف بعد الهيدر. لا يوجد loop داخلي يعيد الملف من البداية بعد آخر صف.

الاستنتاج:

- إذا رأيت بداية الملف مرة ثانية بعد النهاية، فالأرجح أن Run جديد بدأ، وليس أن نفس generator عاد تلقائيًا للبداية.

## فحص مسار تقسيم العمال item workers

الدالة:

`src/core/utils/chunking.py::split_into_chunks`

سلوكها:

- تحول الأصناف إلى list مرة واحدة.
- تقسمها إلى chunks متجاورة.
- لا تعطي كل worker نسخة كاملة من الملف.

الدليل من الكود:

```python
base_size, remainder = divmod(total, effective_workers)
...
chunks.append(materialized[start:start + take])
start += take
```

الاستنتاج:

- استخدام `--item-workers 4` لا يعني أن كل worker يبدأ الملف من الأول.
- لكنه قد يجعل اللوج يبدو مربكًا لأن stdout من workers متعددة قد يظهر بترتيب غير زمني بسبب buffering.

## فحص مسار API/browser fallback

في `src/tawreed/tawreed_bot_api.py`:

- `execution_mode=auto` يحاول API أولًا.
- إذا API غير متاح، يسقط إلى browser.

هذا قد يخلق انطباعًا بأن هناك مرحلتين، لكنه لا يعيد كل الأصناف من البداية إلا إذا بدأ Run كامل جديد أو حصل fallback قبل أي mutation حقيقي.

الاستنتاج:

- هذا سبب محتمل أقل ترجيحًا مقارنة بمشكلة إعادة إرسال Streamlit.

## فحص اسم ملف اللوج

قبل الإصلاح:

`src/ui/order/streamlit_order_form.py::order_output_path`

كان يستخدم:

```python
int(time.time())
```

هذا يعني أن تشغيلين داخل نفس الثانية يمكن أن يأخذا نفس اسم ملف اللوج:

```text
order_output_<seconds>.log
```

الأثر:

- قد يختلط تحليل المشكلة لأن ملف log واحد لا يمثل Run واحدًا دائمًا عند الضغطات السريعة.
- هذا لا يضيف الأصناف وحده، لكنه يخفي الدليل الصحيح ويجعل المشكلة تبدو أغرب.

بعد الإصلاح أصبح يستخدم:

```python
time.time_ns()
```

فتصبح أسماء اللوجات فريدة عمليًا حتى عند تشغيلين متقاربين جدًا.
