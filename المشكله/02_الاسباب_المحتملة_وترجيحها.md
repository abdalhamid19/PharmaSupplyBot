# الأسباب المحتملة وترجيح السبب الأساسي

## السبب الأول: إعادة إرسال نفس Run من Streamlit

الترجيح: عالي جدًا.

الوصف:

واجهة Streamlit كانت تبدأ subprocess جديدًا عند وصول `submitted=True` بدون مقارنة التشغيل الحالي أو السابق ببصمة ثابتة. بعد انتهاء العملية يتم حذف `order_process` من session state، وبذلك يصبح نفس الفورم قادرًا على بدء Run آخر من أول ملف Excel.

لماذا يطابق العرض الذي وصفته:

- المشكلة تظهر بعد أن ينتهي تنزيل الأصناف.
- بعدها يبدأ من جديد، وهذا سلوك Run جديد من أول الملف.
- الكود الخاص بقراءة Excel لا يحتوي على إعادة دوران ذاتية للملف بعد آخر صف.

الإصلاح المنفذ:

- إضافة `order_submission_fingerprint` في `src/ui/order/streamlit_order.py`.
- حفظ البصمة في `st.session_state["last_order_submission_fingerprint"]` بعد بدء العملية.
- رفض تشغيل نفس البصمة مرة أخرى برسالة واضحة.

## السبب الثاني: تداخل/استبدال ملفات log بسبب اسم مبني على الثواني

الترجيح: متوسط.

الوصف:

اسم ملف اللوج كان يعتمد على `int(time.time())`. عند تشغيلين في نفس الثانية، يستخدمان نفس الملف. هذا يجعل قراءة اللوج مضللة وقد يظهر كأن نفس التشغيل أعاد نفسه أو أن الأحداث غير مرتبة.

لماذا يطابق جزئيًا:

- في الملفات الكبيرة قد يحاول المستخدم الضغط أو التحديث أكثر من مرة.
- إذا حدث تشغيلان قريبان، اللوج قد لا يوضح الفصل بينهما بدقة.

الإصلاح المنفذ:

- تغيير اسم اللوج إلى `time.time_ns()` في `src/ui/order/streamlit_order_form.py`.

## السبب الثالث: تشغيل `item_workers` بأكثر من 1

الترجيح: منخفض إلى متوسط.

الوصف:

عند استخدام `--item-workers N` يتم تقسيم الأصناف إلى chunks وتشغيل عدة workers. stdout من عدة processes قد يظهر بترتيب غير متوقع بسبب buffering. قد يبدو للمستخدم أن البداية تكررت أو أن التشغيل عاد من الأول.

لماذا ليس السبب الأساسي المرجح:

- `split_into_chunks` تقسم القائمة إلى أجزاء متجاورة ولا تعطي كل worker كل الملف.
- الدليل من الكود يوضح أن `start` يزيد بعد كل chunk.

ما يجب مراقبته:

- إذا ظهر أول صنف في ملف Excel في summary أكثر من مرة داخل نفس run_id، فهذا سيكون دليلاً على خطأ chunking.
- لم يظهر من الكود الحالي ما يدعم ذلك.

## السبب الرابع: API fallback من API إلى browser

الترجيح: منخفض.

الوصف:

في `execution_mode=auto` يحاول البرنامج استخدام API ثم يسقط للـ browser عند عدم توفر العقد الآمن. إذا فشل API قبل أي إضافة حقيقية، يبدأ browser لنفس الأصناف. هذا fallback مقصود.

لماذا ليس السبب الأساسي المرجح:

- fallback يحدث داخل نفس بداية التشغيل وليس بعد انتهاء كل الأصناف بنجاح.
- عند نجاح API تظهر رسالة `order completed with Tawreed API backend` ولا يتم browser fallback بعدها.

## السبب الخامس: Resume غير مفعل

الترجيح: متوسط كعامل مساعد، وليس أصل المشكلة.

الوصف:

إذا بدأ Run جديد بدون `--resume`، فسيبدأ من أول Excel طبيعيًا حتى لو كانت الأصناف السابقة موجودة في summary.

لماذا مهم:

- إذا حدثت إعادة تشغيل غير مقصودة، عدم تفعيل resume يجعل الضرر أكبر لأنه سيعيد أول الملف.

حل إضافي ممكن:

- تشغيل الطلب من الواجهة مع خيار `Resume from previous summary` عند استكمال تشغيل سابق.
- هذا لا يغني عن حماية منع duplicate submit، لأنه resume يعتمد على summaries وقد لا يغطي كل حالات الإضافة للسلة.

## السبب السادس: ملف Excel نفسه يعيد الصفوف

الترجيح: منخفض جدًا.

الوصف:

قد يكون الملف يحتوي على نفس الأصناف مكررة في آخره، أو به rows مخفية/مفلترة تظهر كأنها بداية ثانية.

لماذا مستبعد حاليًا:

- الفحص أظهر هيدر واحد وبنية بسيطة من 3 أعمدة.
- كود القراءة يقرأ `sheet.iter_rows` خطيًا من بعد الهيدر.

ما يمكن عمله لاحقًا إذا تكررت المشكلة:

- استخراج أول 20 وآخر 20 صفًا من `00000000097.xlsx` ومقارنتها.
- عمل dedupe report للأكواد والأسماء داخل الملف.

## السبب السابع: ضغط زر Run Order مرتين

الترجيح: عالي كآلية تشغيل للسبب الأول.

الوصف:

إذا تم الضغط مرتين أو حدثت إعادة إرسال من المتصفح، فقبل الإصلاح لا توجد حماية تمنع نفس التشغيل.

الإصلاح الحالي يغلق هذا المسار لأنه يقارن بصمة النموذج والملف.

## الترتيب النهائي للأسباب

1. إعادة إرسال نفس Run من Streamlit بسبب غياب بصمة تمنع duplicate submit.
2. ضغط الزر/إعادة إرسال المتصفح كآلية عملية لإطلاق نفس Run مرة ثانية.
3. عدم تفعيل resume كعامل يزيد الأثر عند بدء Run جديد.
4. تداخل اللوجات بسبب اسم مبني على الثواني، كعامل يربك التشخيص.
5. `item_workers` وترتيب stdout، كعامل مظهر وليس سببًا مؤكدًا للتكرار.
6. API fallback، سبب أقل احتمالًا ولا يطابق عبارة "بعد ما ينتهي يبدأ من جديد".
7. عيب في ملف Excel نفسه، الأقل ترجيحًا حسب الفحص الحالي.
